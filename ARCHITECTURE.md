# Архітектура платформи турнірів (Architecture Document)

Цей документ описує технічну архітектуру та структуру платформи для управління турнірами та хакатонами. Проєкт базується на фреймворку Django і слідує монолітній багатошаровій архітектурі.

## 1. Загальний огляд системи

Платформа побудована за паттерном **MVT (Model-View-Template)**, що є стандартом для Django. Вона складається з трьох ключових додатків (apps):
1. **`core`** — Головний застосунок конфігурації. Містить налаштування (`settings.py`), маршрутизатор найвищого рівня та конфігурації для розгортання (WSGI/ASGI).
2. **`tournament`** — Основний бізнес-додаток. Містить моделі для сутностей турніру, завдань, команд, оцінок та сертифікатів. Включає в себе ключові бізнес-сервіси (`services.py`).
3. **`users`** — Додаток, що відповідає за користувачів, їхні ролі, автентифікацію, дозволи (policies), а також містить більшість **Views** (контролерів) та URL-маршрутизаторів для всіх дашбордів системи (адміністратор, організатор, журі, учасник).

## 2. Структура Бази Даних (Models)

Система даних є сильно пов'язаною та нормалізованою. Ключові сутності:

* **`CustomUser`** (`users/models.py`): Розширює стандартну модель користувача Django, додає `role` (participant, jury, organizer, admin) та поля для підтвердження email.
* **`LoginThrottle`**: Система обмеження входу (brute-force захист).
* **`Tournament`**: Центральна сутність звітів. Має життєвий цикл (`draft`, `scheduled`, `registration`, `running`, `finished`).
* **`Team`** & **`Participant`**: Моделі команд та членів. Команда прив'язана до турніру через проміжну модель **`TournamentRegistration`**.
* **`Task`** & **`Submission`**: Завдання турніру та відповідь (рішення) від команди, що містить посилання (GitHub, демо, відео).
* **`JuryAssignment`** & **`Evaluation`**: Призначення окремих членів журі на конкретні "роботи" та їх багатокритеріальне оцінювання (Frontend, Backend, Functionality, UX).
* **`Certificate`** & **`CertificateTemplate`**: Система генерації сертифікатів (переможець/учасник).

## 3. Модель Дозволів (Authorization)

Доступ до `views` жорстко контролюється декораторами та класами з `policies.py`:
1. **Базові рівні доступу**: Залежать від поля `role` у моделі `CustomUser`. Після входу користувача редиректить в залежності від його ролі (наприклад, функція `redirect_by_role`).
2. **Рольові дашборди**: 
    - `admin_dashboard` та група URL з префіксом `/dashboard/` доступні лише для ролей `admin` та `organizer`.
    - `jury_dashboard` доступний для ролі `jury`.
    - `participant_dashboard` доступний для ролі `participant`.
3. **Об'єктні дозволи**: Для редагування команди (`edit_team`) перевіряється, чи є користувач капітаном. Для оцінювання (`submit_evaluation`) перевіряється, чи призначено конкретного журі на конкретну роботу.

## 4. Бізнес-логіка (Services Layer)

Щоб уникнути перенасичення `views.py` (Fat Views) та `models.py` (Fat Models), основна логіка винесена у сервісні класи/функції:
* **`tournament/services.py`**: Містить `RegistrationService` для обробки багатоетапних заявок та `TournamentLifecycleService` для зміни статусів турніру.
* **`users/selectors.py`**: Функції для оптимізованих запитів до бази даних (Database Queries), підготовка контексту для дашбордів. Включає складну логіку генерації лідерборду (`build_tournament_leaderboard`).
* **`users/platform_services.py`** та **`users/team_services.py`**: Сервіси керування платформою та командами (усунення учасників, управління даними).

## 5. Обробка відповідей і файлів (Templates & Static)

* Маршрутизація використовує серверний рендеринг через механізм шаблонів Django (`templates/`). 
* Всі форми прописані у `forms.py` (у `tournament` та `users`) і використовують `django-crispy-forms` для генерації адаптивних UI-елементів.
* Сертифікати рендеряться динамічно (`build_certificate_pdf_response`), де на зображення `CertificateTemplate` накладається текст з використанням бібліотеки `Pillow` (завантаження шрифтів у `load_certificate_font`).
* Статичні файли (CSS, JS, Fonts) обробляються через `WhiteNoise`, як оголошено у `requirements.txt` та налаштовано у `settings.py`.

## 6. Деплоймент (DevOps)

Система підготовлена до публікації як `Infrastructure as Code` (IaC):
* **`render.yaml`**: Описує сервіси для розгортання в Render.com. Встановлює `Web Service` (gunicorn-сервер) та `PostgreSQL` базу даних в рамках безкоштовного тарифу.
* **`build.sh`**: Баш-скрипт, який запускається під час кожного деплою `collectstatic`, накатує міграції `migrate` і (за потреби) створює супер-користувача через змінні середовища (env vars).
* **Змінні середовища (`.env`)**: Безпечні секрети передаються через `.env` (локально) та через секрети інтерфейсу Render (у продакшені). Включають налаштування SMTP або Brevo API для доставки Email-сертифікатів.
